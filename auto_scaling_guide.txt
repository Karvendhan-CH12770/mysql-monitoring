1. access_type
First thing to look at. From best to worst:

const / eq_ref — perfect
ref — good
range — acceptable
index_merge — investigate
index — slow
ALL — critical, fix immediately


2. rows
How many rows MySQL estimates it will scan. Lower is better. High number means the index is not filtering enough.

3. filtered
Percentage of scanned rows that actually pass the WHERE condition.

Above 50% — good
20% to 50% — acceptable
Below 20% — problem, index is not selective enough
Your query — 5.2%, needs improvement


4. possible_keys
Indexes MySQL considered. If this is NULL, no index exists for this query — you need to add one.

5. key
Index MySQL actually chose. If NULL but possible_keys has values, MySQL decided a full table scan was cheaper — investigate why.

6. key_length
How many bytes of the index were used. If much shorter than the full index length, MySQL is only using part of your composite index — column order may be wrong.

7. index_merge
If present, means no single index could handle the query. Usually caused by OR conditions on different columns. Consider redesigning the index or splitting into UNION.

8. using_temporary
MySQL created a hidden temp table. Common with GROUP BY or DISTINCT on non-indexed columns. Hurts performance on large data.

9. using_filesort
MySQL could not use an index for ORDER BY, so it sorted manually. If result set is large this hits disk. Fix by including the ORDER BY column in your index.

10. using_index (Covering Index)
This is the goal. Means MySQL got everything it needs from the index alone — no row lookup needed. Best possible outcome.

11. using_where
MySQL applied a WHERE filter after fetching the row. Means the index alone could not filter completely. Acceptable but check if a better index can push more filtering earlier.

12. using_index_condition
Index Condition Pushdown (ICP). MySQL filters at the index level before fetching the full row. Better than using_where. Good sign.

13. using_join_buffer
A JOIN table has no usable index. MySQL loads rows into memory buffer and loops through them. Add an index on the JOIN column to fix.

14. query_cost
Only visible in FORMAT=JSON. Total estimated cost of the query. Use this to compare before and after adding indexes. Lower is better.

15. attached_condition
The remaining WHERE filter applied after index access. If this is very large (like your 400+ IN list), it means a lot of row-by-row checking is happening after the index scan.
